## 0. 文档与任务管理规则

### 0.1 溯源规则

每份设计文档必须包含：

```text
源头文档：项目介绍.txt
溯源段落：
- 技术架构：...
- 项目实现：...
覆盖任务：
- Txx ...
```

每个开发任务必须能反查到：

1. `项目介绍.txt` 的需求来源。
2. 对应模块设计文档。
3. 对应接口、数据表、状态机或页面设计。
4. 对应测试用例。

### 0.2 建议设计文档目录

```text
docs/
```

验收标准：

- 用户输入“重启/删除/修改配置/升级/下发规则”会暂停流程。
- 终端打印 review_id、风险原因、建议动作、恢复入口。
- 前端和 API 都能看到 pending review。
- approve 后从 checkpoint 继续，不重新执行整轮 chat。
- reject 后流程恢复并给出终止说明或替代建议。
- resume token 只能消费一次。

## 11. 推荐开发顺序总览

```text
设计基线
  → 后端骨架
  → PostgreSQL / Redis / Milvus
  → Harness 生命周期
  → Tools / Adapters
  → Agent Runtime / Sub-agents / Skills
  → Risk interrupt / Review resume
  → 前端工作台
  → 多模态图片研判
  → 审计与验收
```

首个可交付里程碑：

```text
用户问“北门摄像头离线怎么排查”
  → /chat
  → Harness 路由到 ops-troubleshooter
  → 查询设备状态
  → 检索 SOP
  → 生成任务清单
  → 返回 answer/evidence/trace/tasks
  → PostgreSQL + Redis + Audit 可追溯
```

第二个可交付里程碑：

```text
用户说“重启北门流媒体服务”
  → RiskGate 命中高风险
  → Agent interrupt
  → 终端输出 [REVIEW_REQUIRED]
  → 前端展示待审核
  → 人工 approve
  → LangGraph resume
  → 返回恢复后的处理结果
```
