# 07 风险控制与中断恢复设计

源头文档：`项目介绍.txt`

溯源段落：
- 高风险动作会触发中断、终端提示和人工确认，审批后从检查点恢复并继续原流程。
- 审批通过后通过 LangGraph interrupt / resume 从 checkpoint 继续执行原流程，审批拒绝后恢复流程并给出终止说明或替代建议。

覆盖任务：
- T22 ReviewStore
- T25 Redis checkpoint
- T45 request_human_review
- T60 RiskGate
- T61 InterruptManager
- T62 review 状态机
- T63 终端提示
- T64 Review API
- T65 interrupt / resume
- T66 拒绝后的替代建议

---

## 1. 风险来源

- 关键词：重启、删除、修改配置、升级、下发规则、关闭告警、执行命令。
- 工具标记：`requires_review=true`。
- 对象范围：生产服务、核心设备、批量设备。

---

## 2. 中断流程

```text
高风险动作
  → RiskGate high
  → InterruptManager.pause_for_review
  → LangGraph interrupt
  → Redis checkpoint
  → PostgreSQL review_request pending
  → terminal [REVIEW_REQUIRED]
  → ChatResponse interrupted=true
```

---

## 3. 恢复流程

```text
approve/reject
  → ReviewStore 记录人工决定
  → InterruptManager.resume_after_review
  → graph.invoke(Command(resume=...))
  → ResponsePersister 保存后续结果
```

拒绝时也要 resume，让 Agent 输出终止说明或替代建议。

---

## 4. 终端提示

```text
[REVIEW_REQUIRED]
review_id=...
thread_id=...
risk_level=high
reason=...
proposed_action=...
resume_api=POST /reviews/{review_id}/resume
```

不得输出密钥、连接串、完整图片 base64。

---

## 5. Review 状态机

```text
pending
  ├─ approved → resumed → completed
  ├─ rejected → resumed → cancelled
  └─ expired
```

---

## 6. 验收

- 高风险动作审批前不执行。
- 终端和前端都能看到待确认信息。
- approve 后从 checkpoint 继续。
- reject 后给出终止说明或替代建议。
- resume_token 只能消费一次。

---

## 7. 当前实现状态

- 已实现 `RiskGate`，命中重启、删除、修改配置、升级、下发、关闭告警、执行命令等关键词时要求审核。
- 已实现 `InterruptManager.pause_for_review()`：创建 pending review、写 `checkpoint_ref` 占位、输出终端 `[REVIEW_REQUIRED]`。
- 已实现 `InterruptManager.resume_after_review()`：approved 进入 `completed`，rejected 进入 `cancelled`，并写 assistant 恢复消息。
- 已实现 `/reviews/{id}/approve`、`/reject`、`/resume` API 骨架。
- 尚未接入 LangGraph `Command(resume=...)` 和 Redis checkpoint；当前恢复是状态机骨架，真实图恢复在 T65 完成。
